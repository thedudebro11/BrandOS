## Phase 5 — Platform Consolidation (2026-07-08)

Engineering-quality phase, not a feature phase — deliberately trimmed by explicit user direction to four objectives: shared type system, multi-workspace verification, an honest architecture/code audit, and a platform validation report. Explicitly out of scope this phase: API client generation, developer-doc expansion, lint/format tooling, cosmetic refactors, speculative optimization.

- **Shared type system** (`app/shared-types/index.ts`, new): the single source of truth for every API request/response shape (`WorkspaceSummary`, `OverviewData`, `CaseSummary`/`CaseDetail`/`CaseTemplate`/`CaseEvidenceSuggestion`, `AssetSummary`/`AssetIntelligence`/`AssetFacets`, `PriorityOfUseData`, `SearchResult`, `ActivityEvent`, `ActionItem`, `ReviewQueueEntry`, `DuplicateGroup`). Both `app/src/api/routes.ts` (every handler now constructs an explicitly-typed response object) and `app/web/src/api.ts` (re-exports from shared-types instead of declaring its own copies) import from here — the exact fix for the type-duplication risk documented in Phase 4 and confirmed as a real incident in Phase 4.5. Wired via a plain relative-import folder (no build step, no new package.json, no npm workspaces) — `app/tsconfig.json`'s `include` and `app/web/vite.config.ts`'s `server.fs.allow` both updated to reach the shared location. Verified end-to-end: both packages typecheck clean, frontend production build unchanged in size (confirms type-only imports are fully erased, zero runtime cost), and the Vite dev server correctly serves the shared file (`fs.allow` fix confirmed via direct HTTP request, not assumed).
- **Multi-workspace isolation, finally verified for real** (deferred across Phases 3.5, 4, and 4.5): a new automated test suite (`tests/core/multi-workspace-isolation.test.ts`, 5 tests) proves two independently-scanned fixture workspaces share nothing — separate database files, zero asset leakage, zero case leakage, zero search-result leakage, and correctly independent duplicate-group/review-queue detection even when both workspaces contain byte-identical content (the scenario most likely to falsely merge them if isolation were broken). Beyond fixtures: PrecisionWorkz was scanned and exercised through the **actual API layer for the first time** — real HTTP requests against a live server confirmed its overview, cases, and search all correctly return empty/isolated results while Fatletic's real 5 cases and 198 real search results remained simultaneously unaffected in the same running process (the real test of `db-cache`'s per-workspace connection isolation, not just a theoretical one).
- **Architecture & code audit** — real findings, documented not auto-fixed (per explicit instruction): duplicated row-mapping logic (`mapAssetRow` in `repositories.ts` and `mapAsset` in `query-engine.ts` independently implement the same snake_case→camelCase conversion), two genuinely dead functions (`CaseBuilderService.createCustom()` and `import-engine.ts`'s `watchWorkspace()`, both zero-usages-found outside their own definitions), and four scaffolding directories empty since Phase 0/1b with zero content across 5+ phases (`app/src/plugins/`, `app/shared/`, `app/scripts/`, `app/obsidian-temple/`). Also confirmed clean: no unused npm dependencies, no circular dependencies between `core/` and `api/`, no workspace/brand literal leakage in any TypeScript source (only a factual code-comment reference), all 5 CLI scripts correctly wired to `package.json`. Full prioritized findings in `PHASE_5_PLATFORM_VALIDATION_REPORT.md`.
- **Tests:** 111 total (up from 106), all passing. Both packages typecheck and build clean throughout.
- **Docs:** this entry, `ARCHITECTURE_REVIEW.md` Phase 5 section, `ARCHITECTURE_DECISIONS.md` Phase 5 consistency note (no new ADR — Section 1 implements existing principles, doesn't change them), `IMPLEMENTATION_PLAN.md`, and `PHASE_5_PLATFORM_VALIDATION_REPORT.md` (new).

## Phase 4.5 — Mission Control Evolution (2026-07-08)

Explicit rule carried forward and re-verified: the dashboard still contains no business logic. Every new value traces to an existing or newly-added *engine-layer* function; nothing is computed in a route handler or a React component.

- **New engine capabilities (small, defensible, core layer):**
  - `activity-feed-engine.ts` — a read-time UNION over `scan_runs`, `knowledge_validation_runs`, `cases`, `case_links`, `relationships`, and `data_health_findings`, producing one chronological feed. No new storage.
  - `QueryEngine.listAssetsFiltered()` — general-purpose classification/tag/extension/needs-review filtering + sort, the Evidence Explorer's data source, written to be reusable by any future consumer.
  - `CaseBuilderService.suggestUnlinkedEvidence()` — groups unlinked assets by tag for empty-state guidance, explicitly framed as browse candidates, never a claim that they support the case (ARCHITECTURE_PRINCIPLES.md #4).
- **New API routes:** `/activity`, `/action-center` (pure composition of existing counts/gaps into a clickable list), `/review-queue` (enriched drill-down), `/duplicates` (enriched drill-down), `/cases/:id/suggestions`, and `/assets` upgraded to support real filtering + facet counts.
- **Information hierarchy (Overview redesign):** hero tier (Trademark Readiness, Priority of Use, Needs Review, Workspace Status, Evidence Quality) rendered large; mid tier (Timeline, Relationships, Cases, Health) at normal weight; low tier (Architecture Health, Duplicate Groups) small and quiet at the page bottom — via typography/sizing/layout only, no new data.
- **New pages:** Asset Detail (closes the gap flagged in Phase 4's own review — fills in classification, tags, a relationship "knowledge navigation" panel, timeline, metadata, linked cases, hash), Needs Review drill-down, Duplicate Groups drill-down, Evidence Explorer (filter by classification/tag, sort, with real facet counts).
- **Case Workspace enrichment:** Readiness/Confidence hero row, empty-state evidence suggestions (tag-grouped, neutral framing), honest Notes placeholder ("not yet supported" — no fake feature), honest Exports placeholder ("planned for a future phase").
- **Search grouping (Section 7):** Cmd+K results now grouped by entity type (Cases, Assets, Timeline Events, Tags) in priority order; selecting an asset result now correctly opens its real Asset Detail page.
- **Two real bugs found and fixed during this phase, not shipped silently:**
  1. Search results for assets returned only the internal numeric row id, which the new Asset Detail route can't use (it keys on the stable `AST-########` string id) — `SearchResult` gained a `stringId` field, populated for every asset-producing query.
  2. `QueryEngine.listAssetsFiltered()`'s initial version could reference a SQL alias (`cl.confidence`) that wasn't always joined when sorting by confidence without also filtering by classification/review status — fixed by always joining classifications (as a `LEFT JOIN`) whenever any of the three conditions need it.
- **Verification, with a caveat repeated from Phase 4:** both packages typecheck and build cleanly; all 106 backend tests still pass; every new API route was exercised with real `curl` calls against Fatletic's actual data (case suggestions, asset detail with a real relationship chain, filtered asset queries, activity feed, action center). One of those verification passes caught a stale background server process serving pre-fix code — confirmed and corrected, noted here as a real environment quirk (background process lifecycle in this sandboxed session is not fully reliable across tool calls) rather than swept under the rug. As before, no browser-automation tool exists here, so visual rendering itself is still unverified by screenshot.

## Phase 4 — Mission Control (2026-07-07)

First UI phase. Hard rule: the dashboard contains no business logic — every visualization is `Dashboard → Query Engine → Knowledge Layer → Database → Core Engine`, never a side path. First code in the project outside `app/src/core`.

- **Architecture decision (ADR-011, explicitly approved):** "Local Web App Now, Desktop Shell Later." Confirmed no Rust/cargo toolchain exists in this environment (same class of blocker as ADR-009's `better-sqlite3` finding) — Tauri (ADR-006's preference) cannot compile here. Serving Mission Control as a local web app instead: a small Node/Express API plus a Vite/React frontend, structured so a desktop shell can wrap it later without a frontend rewrite.
- **API layer** (`app/src/api/`): `db-cache.ts` (one cached `WorkspaceDatabase` per workspace, explicit `save()` after mutations — the CLI's save discipline extended to a long-running server), `routes.ts` (every handler a thin pass-through to an existing engine function — `QueryEngine`, `CaseBuilderService`, `AssetIntelligence`, `EvidenceEngine`, `computeEvidenceQualityMetrics`, `search`), `server.ts` (Express app, CORS, static-serving the built frontend).
- **One new engine-layer function**, deliberately placed there and not in the API/dashboard: `bandLabel()` (High/Medium/Low score banding) added to `evidence-quality-metrics.ts` so any future consumer gets identical cutoffs, not just Mission Control.
- **Frontend** (`app/web/`, a separate Vite + React + TypeScript package): design direction is an "instrument panel for evidence," not a marketing site — IBM Plex Sans for narrative, IBM Plex Mono for fact (Asset IDs, hashes, dates, scores), a graphite/amber palette with a three-color confidence semantic (strong/weak/conflicting) used as the recurring signature element (`ConfidenceBar`) across tiles, case rows, and asset rows. Cmd+K command palette (Raycast/Linear-inspired) backed by the real Search Engine. Pages: Overview (the exact metrics layout specified), Cases (list + detail with all 10 required sections — Executive Summary, Evidence Overview, Timeline, Supporting Assets, Related Cases, Confidence, Risks, Missing Evidence, Conflicts, Recent Changes, Linked Documentation), Priority of Use (the full chain view — numbered steps justified here since the chain is a real reasoning sequence, not decoration).
- **One new, honest repo function:** `listRelatedCases()` — cases sharing a linked asset — pure data retrieval, no judgment.
- **Fatletic's 5 named cases seeded** (`npm run seed-cases`) via the existing `CaseBuilderService`, as empty shells — deliberately **not** auto-linked to any assets. Auto-linking evidence to a named legal case is a materially stronger claim than tagging a photo "Product Photo," and nothing has reviewed that linkage yet; each case starts honestly empty rather than presenting invented curation.
- **Closed out documented Phase 3.5 debt before it became a headline dashboard number:** the health-score formula flagged as "measurably broken" in `PHASE_3_5_VALIDATION_REPORT.md` was recalibrated — percentage-of-assets-affected instead of flat finding-count subtraction, blended 50/50 with Knowledge Validation pass rate. Verified against real Fatletic data: 0/100 (broken formula) → 99/100 (fixed formula, one real warning-level finding out of 199 assets).
- **Real end-to-end verification, not just unit tests:** ran the actual API server and Vite dev server together, confirmed the dev proxy correctly forwards `/api` calls, and exercised a live mutation (`POST .../cases/:id/links`) that correctly re-triggered `CaseBuilderService`/`EvidenceEngine` recomputation (case evidence strength 0→100 after linking one high-confidence asset) — then reverted that test link so it wouldn't sit in real Fatletic data looking like genuine curation.
- **Regression test added and one existing test corrected** for the health-score formula change (the old test asserted the *old, broken* formula's behavior — an info-only finding no longer lowers the score, which is the fix working as intended, not a regression). 106 tests total (up from 105), all passing; both `app/` (backend) and `app/web/` (frontend) build and typecheck cleanly.
- **Testing ceiling, disclosed honestly:** no browser-automation tool is available in this environment, so rendering was verified via successful production build (`vite build`), strict typecheck, and real API responses matching the frontend's expected shapes — not a visual/screenshot check. Worth a manual look in an actual browser before treating the visual design as final.

## Phase 3.5 — Evidence Reliability & Knowledge Validation (2026-07-07)

No UI, dashboard, Obsidian, reports, AI, exports, or reorganization — purely correctness/trustworthiness work on what already existed. Fixes the Phase 3 epoch-timestamp finding architecturally (ADR-010), not with a patch.

- **Schema:** `migrations/0003_evidence_reliability.sql` — 6 new tables (`candidate_dates`, `date_source_priorities` seeded with the specified default order, `resolved_dates`, `data_health_findings`, `knowledge_validation_runs`) plus explainability columns added to `classifications` and `review_queue`.
- **Timeline Intelligence** (`services/timeline-intelligence/`): collects candidates from filesystem, EXIF/PDF/PSD/XCF metadata, filename patterns, folder patterns, and relationship inference; plausibility-checks every one (epoch/pre-1990/future rejected, never deleted, always explained why); the Resolution Engine picks a winner by configurable priority among plausible candidates only, with full reasoning and every rejected alternative recorded.
- **Evidence Provenance Engine**: traces a resolved date through its winning candidate to a hash-verified original asset and the workspace — the concrete, currently-buildable slice of the full Trademark-Readiness-to-hash chain described in the spec.
- **Classification improvements**: new known-product-folder-structure rule (real signal from Fatletic's actual "Clothing Promo" folder tree), sibling-majority context boost for the generic low-confidence "Image" fallback — both deterministic, no AI.
- **Needs Review Intelligence**: every review-queue entry now carries a concrete suggested action, estimated effort, and potential impact, computed from real duplicate/relationship/case-link counts, not invented.
- **Data Health Engine**: new epoch/implausible-date, broken-candidate-provenance, and duplicate-metadata checks — delegates to Phase 3's Integrity Engine for the checks that already existed rather than reimplementing them.
- **Knowledge Validation Engine**: 9-point pass/fail checklist (every asset has an ID, every relationship/timeline/case-link/report references a real asset, every active asset is hashed, every resolved date references a real candidate, no duplicated metadata, provenance chains complete).
- **Evidence Quality Metrics**: 11 named, formula-documented metrics (timeline/metadata/relationship/classification completeness, confidence distribution, needs-review %, duplicate/evidence/provenance coverage, health score, missing/conflicting evidence scores) — one reusable function, not one-off queries per future consumer.
- **Wired into the scan pipeline:** candidate collection + date resolution run on every new/changed asset, plus a relationship-derived second pass (an export can inherit a candidate from its resolved source file). Classification/tagging upgraded to call the new explainability path.
- **CLI:** `analyze` extended with Data Health, Knowledge Validation, and Evidence Quality Metrics output (replacing the plain Phase 3 integrity-check section, which is now folded into Data Health rather than run twice).
- **Tests:** 30 new tests (105 total, up from 75) across 5 new files. Two real bugs found and fixed via testing: an ambiguous `asset_id` column reference in two SQL joins (the `assets` table has both an integer `id` and a string `asset_id` column — a genuine SQL bug, not a test artifact), and a resolution-engine gap where re-resolving to "no plausible candidates" left a stale prior resolution in place instead of clearing it.
- **Real run against Fatletic** (199 real assets, fresh scan): confirmed all 199 `filesystem_created` candidates correctly excluded as epoch artifacts; system fell through to `filesystem_modified` (185), real PDF metadata (11), and EXIF (3); average resolved confidence 52.1/100 (honest, not the prior false 100). Classification completeness rose from 36% (Phase 3) to 73%, with the generic "Image" bucket dropping from 128 to 53 assets as 50 correctly reclassified to a new "Product Photo" category and Export detection improved from 3 to 26. Full numbers in `app/docs/PHASE_3_5_VALIDATION_REPORT.md`.
- **New finding, disclosed not hidden:** continuous-use score remains weak (1/100) even at 100% timeline completeness, because the `filesystem_modified` fallback — while plausible — isn't independently verified as accurate for continuous-use purposes; and the health-score formula proved miscalibrated on real data (hit 0/100 from finding *volume*, not severity). Both documented as open items, not smoothed over.

## Phase 3 — Knowledge Layer (2026-07-07)

No UI, no dashboard, no Obsidian generation, no reports, no AI classification — all explicitly out of scope and not built. Added `app/docs/ARCHITECTURE_PRINCIPLES.md` (the project's constitution) and `app/docs/ARCHITECTURE_REVIEW.md` (new, cumulative, updated after every phase from here on).

- **Schema:** `migrations/0002_knowledge_layer.sql` — 9 new tables (`tags`, `asset_tags`, `classifications`, `review_queue`, `evidence_assessments`, `evidence_gaps`, `case_templates` seeded with the 8 named templates, `asset_notes`, `integrity_checks`). Every new table references `assets.id` or another table's id — nothing duplicates a fact that lives elsewhere.
- **Rule-based classification** (`services/classification-engine/`): deterministic rules, no AI. Path-context rules (Instagram/Printful/proof/invoice/customer/legal keywords) run before extension rules, since folder context is a stronger signal than file type alone. Confidence reflects real signal strength — a PSD's own binary header scores 95, a bare image with no relationship or path context scores 65 and lands in Needs Review.
- **Tag engine** (`services/tag-engine/`): deterministic tag suggestions from classification category + path keywords, every suggestion carries a stated reason.
- **Evidence engine** (`services/evidence-engine/`): completeness, continuous-use, and priority-of-use scores computed by plainly-documented formulas over real classification/timeline rows — every assessment's `notes` field states exactly which counts produced the score. Detects and records evidence gaps (e.g. "no dated evidence for Marketing Evidence category").
- **Case Builder** (`services/case-builder/`): full service — create from template or custom, link/unlink assets and timeline events by reference only, case-scoped evidence-strength assessment (delegates to the evidence engine, no separate scoring logic).
- **Query engine** (`services/query-engine/`): discrete typed methods (first-by-category, every-by-extension, before-date, connected-to, missing-metadata, duplicate-groups, orphaned, by-tag, needing-review) — a reusable foundation, not a NL query parser (explicitly future scope).
- **Graph engine** (`services/graph-engine/`): a read-time view over relationships/case_links/asset_tags/timeline_events — no separate graph storage, per Principle #8.
- **Provenance engine** (`services/provenance-engine/`): cycle-safe BFS both directions over the relationships graph, builds an upstream→self→downstream chain for any asset.
- **Integrity engine** (`services/integrity-engine/`): detects broken references, hash mismatches, missing/duplicate/orphaned assets, circular relationships, all persisted to `integrity_checks`.
- **Search foundation** (`services/search-engine/`): unified LIKE-based search across filenames, paths, asset IDs, hashes, tags, metadata values, case titles, timeline titles — explicitly not FTS5/semantic yet (future scope), but swappable behind one function without touching callers.
- **Asset Intelligence** (`services/asset-intelligence/`): the aggregator — composes classification, tags, relationships, timeline, provenance, linked cases, and notes for one asset by Asset ID, entirely from existing normalized tables (no new asset columns, no duplicated storage).
- **Wired into the scan pipeline:** `import-engine.ts` now classifies and tags every new/changed asset after relationship detection (so "is this image an Export?" can see real relationship data), plus catches up any asset from a prior scan that has no classification yet. Four new events (`classification.assigned`, `tags.assigned`, `evidence.assessed`, `integrity.issue_found`).
- **CLI:** `npm run analyze -- <workspaceId>` (runs evidence assessment + integrity check, prints a summary).
- **Tests:** 42 new tests (75 total, up from 33) across 6 new files covering every engine, all against real scan output (not mocked data).
- **Real run against Fatletic** (199 real assets): 128 Image, 25 Design Source, 18 Documentation, 16 Commerce Evidence, 9 Historical Evidence, 3 Export. Completeness 36/100 (71 of 199 confidently classified — the rest correctly land in Needs Review rather than being guessed). Tag counts: Brand/Logo 109 each, Final 80, Product 61, Evidence 25, Prototype 25, Historical 9, Invoice 7, Unused 5. Integrity check: 58 issues, all info-level (11 duplicate groups, 47 orphaned assets) — zero critical/warning issues (no broken references, no hash mismatches, no circular relationships).
- **Significant finding, not a code bug:** every one of the 199 timeline events (100%) came back with a `file_created` date of exactly `1970-01-01` (Unix epoch). This tanked the continuous-use score to 1/100. Root cause: WSL's `drvfs` mount (mounting the Windows C: drive into Linux) does not reliably populate `birthtime` — Node's `fs.Stats.birthtime` falls back to epoch zero. This is an environment/filesystem limitation, not a BrandOS bug, but it means **`created_at` is currently not a trustworthy evidence signal on this setup** — a real risk for a system whose whole purpose includes Priority of Use dating. Documented in full in `ARCHITECTURE_REVIEW.md` with a recommended fix (detect epoch-or-near-epoch birthtime and fall back to `modified_at`, flagged as inferred not verified) for a future phase — not fixed in this one, since it's a Phase 2 ingestion concern outside Phase 3's declared knowledge-layer scope, and is disclosed here rather than silently patched.

## Phase 2 — Core Engine (2026-07-07)

Read-only engine only: no UI, no Obsidian generation, no report generation, no AI classification, no evidence reorganization — all explicitly out of scope and not built. First phase to write actual application code.

- **Driver change:** `better-sqlite3` (originally assumed by ADR-001/006) fails to install in this environment — no C++ toolchain, and Node v20.20.2 predates the built-in `node:sqlite`. Switched to `sql.js` (WASM SQLite), no native compilation required. Documented as ADR-009.
- **Project scaffold:** `app/package.json`, `tsconfig.json`, `vitest.config.ts`; dependencies `sql.js`, `exifr`, `pdf-parse`, `ajv`; dev dependencies `typescript`, `tsx`, `vitest`, `@types/node`, `@types/sql.js`. Installed cleanly, no native builds.
- **Database:** `app/src/core/db/` — `connection.ts` (sql.js wrapper: run/get/all/exec/transaction/save), `migrate.ts`, `migrations/0001_init.sql` (14 tables + `_migrations`), `repositories.ts` (typed data access, including the snake_case-DB-column → camelCase-TS-type mapping every read goes through).
- **Workspace layer:** `app/src/core/workspace/` — `workspace.schema.json` (the JSON Schema deferred since Phase 1b), `workspace-config.ts` (ajv validation, id-must-match-folder-name check), `workspace-registry.ts` (multi-workspace discovery, skips invalid workspaces without crashing the whole registry).
- **Filesystem guard:** `app/src/core/fs/workspace-fs.ts` — `WorkspaceFs` can read anywhere under a workspace but its only write method (`writeGenerated`) is hard-restricted to `.brandos/`; there is no rename/delete method anywhere in the engine.
- **Scanner, hasher, metadata:** `app/src/core/services/{file-scanner,hashing-engine,metadata-engine}/` — recursive read-only scan with ignore rules (also excludes `workspace.json`/root `README.md` as platform config, not evidence); streaming SHA-256 + duplicate-group detection + `FILE_MANIFEST.json`/`.csv` + `CHAIN_OF_CUSTODY.md` generation; metadata extraction for images (EXIF/IPTC/XMP), PDF, PSD and XCF (hand-rolled header parsers, no dependency), SVG, AI, JSON, Markdown, text, and best-effort video (ffprobe if present).
- **Asset IDs, relationships, timeline:** permanent `AST-########` IDs (never reassigned); one basic same-stem source→export relationship heuristic; timeline events from filesystem dates + a generic `content_date` bucket from metadata.
- **Import engine:** `app/src/core/services/import-engine/import-engine.ts` — orchestrates scan → hash → metadata → asset upsert → duplicate check → relationships → timeline → manifests, incrementally (unchanged files skip re-hash), read-only throughout. Includes a best-effort polling `watchWorkspace()`.
- **Event bus + logging:** `app/src/core/events/event-bus.ts` (typed pub/sub, full planned event vocabulary including foundation-only events for cases/dashboard/Obsidian that nothing produces yet) and `app/src/core/logging/logger.ts` (every scan/hash/write/skip/error → `logs` table + console).
- **CLI:** `npm run scan -- <workspaceId>`, `npm run verify -- <workspaceId> [assetId]`, `npm run list-workspaces` (all via `tsx`, no build step needed for dev use).
- **Tests:** 33 passing (vitest) across 7 files — config loading (against the real Fatletic and PrecisionWorkz configs), hashing, scanning/ignore rules, every metadata extractor (including synthetic-binary PSD/XCF fixtures built byte-by-byte in the test itself), migrations (including persist-and-reload), the event bus, and 7 full import-engine integration scenarios.
- **Bugs found and fixed during this phase** (via typecheck + real testing, not left in): sql.js rows are snake_case, not camelCase — every `AssetRecord` read now goes through an explicit mapper; the workspace-root path in an early test had one extra `../`; `workspace.json`/`README.md` were briefly being scanned as if they were evidence; the scan summary's "duplicate groups found" counter was counting per-asset registration events (13) rather than distinct groups (11, confirmed by direct query) — fixed to query the true count.
- **Real integration proof:** ran the actual CLI scan against `workspaces/Fatletic/` (199 real files — logos, PSD/XCF/AI source files, PDFs, product mockups, promo video/images, business text docs) and `workspaces/PrecisionWorkz/` (0 files, config-only stub, proving the engine runs against a workspace with every module inactive). Fatletic result: 199 assets created, 11 duplicate groups (matches the near-duplicate logo variants flagged in Phase 0's analysis), 3 source→export relationships detected, 0 errors. Verified afterward: original evidence tree byte-for-byte untouched (`git status` shows no new diffs on any evidence file), all generated output confined to `workspaces/Fatletic/.brandos/` (correctly gitignored).
- **`app/.claude`:** deleted (`git rm`), per explicit user confirmation following the Phase 1b investigation.
- Known minor cosmetic note: on this case-insensitive filesystem (WSL/drvfs), CLI output echoes whatever casing was typed for the workspace ID (e.g. `fatletic` vs the real `Fatletic` folder) in printed paths; verified this resolves to the same physical directory (same inode), not a duplicate — no data-safety impact, but worth normalizing to canonical casing in a later pass.
- Also noted, not yet fixed (out of scope for Phase 2): `app/obsidian-temple/` (existing empty scaffold folder) is misspelled relative to the `app/obsidian-template/` location named in Phase 1b's approved Obsidian decisions — needs reconciling when Phase 7 (Obsidian) starts, not before.

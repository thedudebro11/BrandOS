# Changelog

Human-readable log of what happened in each approved phase. See `IMPLEMENTATION_PLAN.md` for the forward-looking plan and `ARCHITECTURE_DECISIONS.md` for the reasoning behind specific decisions, and `ARCHITECTURE_REVIEW.md` for the cumulative retrospective updated after every phase.

## Phase 9 — Visual Knowledge Graph & Timeline Explorer (2026-07-08)

Full spec, approved after a scoping check. The single hard rule held throughout: the visual layer exposes existing engine data and never implements its own logic — every new UI page is a thin renderer over a thin API route over an existing (or newly-generic) engine function.

- **Knowledge Graph extended to all 9 node types**: `graph-engine.ts`'s `buildGraph()` grew from 4 node types (asset/case/tag/timeline_event) to 9 (+ workspace, evidence, report, obsidian_note, plugin), with real edges for every new type — workspace→case/plugin/report, case→report (deduped against the pre-existing `case_links` report edge), case/workspace→evidence (latest assessment per scope+dimension only, not the full append-only history), obsidian_note→its documented entity. Verified against real Fatletic data: 1,061 real nodes, 1,274 real edges, all 9 types populated (including discovering and fixing, along the way, that the live Fatletic `obsidian_notes` table had never actually persisted — regenerating the vault populated it for real).
- **Path Discovery** (`path-discovery.ts`): one breadth-first search, parameterized by 6 structural edge-filters (shortest/evidence/relationship/timeline/case/dependency) — never 6 separate traversal implementations. `dependency` is the one directional kind (asset→asset only in the recorded source→export direction); verified with a real forward-succeeds/reverse-fails test against Fatletic's actual relationship data.
- **Evidence Path Explorer backend** (`evidence-path.ts`): full breadth-first reachability trace from one asset (not just point-to-point), reusing the same edge-filter discipline as Path Discovery's `evidence` kind.
- **Node Inspector** (`node-inspector.ts`): `getNodeDetail()` composes existing engines per node type — `getAssetIntelligence()` for assets, `composeCaseDetail()` for cases, direct table reads formatted for display everywhere else. Zero new scoring or inference logic.
- **Timeline Explorer backend** (`timeline-explorer.ts`): chronological, filterable, year-grouped view over Phase 3.5's resolved dates — the same "engine's actual answer," never a raw filesystem timestamp, every other consumer already uses.
- **ADR-007 Safe Citation Mode, implemented** (the phase's special requirement, explicitly gating "before AI or public-facing exports build on the report system"): a single, generator-agnostic implementation point (`safe-citation.ts`), called once by `report-generator.ts` right after `definition.generate()` — none of the 9 report generators from Phase 8 needed to change or even know this exists. Default `citationMode: "safe"`; every real filename/path is replaced with a sequential `Exhibit A-N` label in every narrative section, Asset IDs are left untouched (per ADR-007's "reference Asset IDs by default"), and a clearly-labeled "Supporting Evidence Index (Internal — Do Not Distribute)" section preserves full traceability within the same document. `citationMode: "full"` (CLI: `--full`) opts out explicitly.
  - **A real gap found and fixed during implementation, not after**: the first version scoped the redaction dictionary to a report's own `citationIndex` and missed a real leak — Evidence Binder's Chain of Custody section embeds `traceResolvedDateProvenance()`'s "Cross-Referenced Source Asset" line, which can name a *different* asset than the one being cited and was never independently added as a Citation. Fixed by scanning every active asset's real filename/path against the report's actual rendered text instead of trusting the citation list to be complete — caught by generating a real report against real Fatletic data and grep-checking the output, not by the test suite (written after, informed by this exact finding).
  - Verified: zero filename/path leakage in any narrative section across all 9 report types against real Fatletic data; deterministic (regenerating unchanged data produces an identical content hash and identical exhibit numbering); fully reproducible (every exhibit-labeled citation still resolves to a real Asset ID).
- **Frontend**: added `cytoscape` (canvas-rendered, pure JS, no native compilation — checked against the same constraint that has governed every prior dependency choice since ADR-009). Three new Mission Control pages — Knowledge Graph (`GraphCanvas.tsx` + `KnowledgeGraph.tsx`: zoom/pan/expand/collapse/filter-by-type/search/path-highlighting, styled to the existing design tokens), Timeline Explorer, Evidence Path Explorer — plus a reusable `NodeInspectorPanel.tsx` (Section 5's inspection fields). "Lazy loading / incremental expansion" (Section 7) is real, not cosmetic: the full graph is fetched once, but only hub-type nodes (workspace/case/report/plugin/evidence) render by default, with the ~1,000 asset/tag/timeline_event/obsidian_note nodes added to the visible set only as a user expands a node's neighbors.
- **Verification**: both packages typecheck and build clean; 22 new tests (15 graph/path/inspector/timeline, 7 safe-citation) — 178 total, all passing; every new API endpoint exercised live against real Fatletic data via curl; every new frontend module confirmed to transform cleanly through Vite's dev server (module-level verification — no browser-based visual verification was possible in this environment, stated plainly rather than implied).
- **Not built, documented rather than hidden**: no true virtualized/windowed rendering for extremely large graphs (the default hub-only view plus on-demand expansion is the real mitigation at Fatletic's current ~1,000-node scale; a canvas-level viewport-culling layer would need revisiting at a materially larger scale); search in the Knowledge Graph filters/highlights only among currently-visible nodes, not the full unexpanded graph; no automated visual/browser-based UI test exists anywhere in this project yet (a standing gap, not new to this phase).

## Phase 8 — Professional Reports & Evidence Binders (2026-07-08)

Full spec, approved after a scoping check: all 9 report types, all 4 output formats each. Hard rule enforced throughout: reports consume existing engines only — no report-specific scoring, no fabricated claims, every factual statement carries a citation.

- **Schema:** `migrations/0006_reports.sql` — `reports` (one row per generated report: type, scope, version, content hash, 4 file paths). `case_links.linked_type = 'report'` has existed since Phase 3's schema; this is the first phase to produce report rows it can actually reference.
- **Report data model** (`core/services/report-engine/report-types.ts`): one `ReportData` shape (sections, citations, a deduped citation index/appendix) every report type produces and every renderer/validator consumes — the "single data model, multiple renderers" design. A `Citation` carries `assetId`/`timelineEventId`/`caseId`, `confidence`, `sourceType`, and — when the Obsidian vault (Phase 6) has been generated — a real, checked `obsidianNotePath` cross-reference.
- **Report registry** (`report-registry.ts`): all 9 report types (`trademark_readiness`, `priority_of_use_dossier`, `evidence_binder`, `brand_history`, `case_summary`, `missing_evidence`, `needs_review`, `duplicate_assets`, `workspace_health`) as `ReportDefinition` entries. The generator never branches on report type — adding a 10th report means one registry entry and one file under `reports/`, nothing else touched.
- **Report generator** (`report-generator.ts`): compose → validate → render (Markdown/HTML/PDF-ready-HTML/JSON) → write (new guarded `WorkspaceFs.writeExport()` surface, honoring `workspace.json`'s existing `paths.exports` field) → persist a `reports` row. Deterministic content hashing (`hashReportContent()`) excludes `generatedAt` and uses a stable, key-sorted stringify — proven, not assumed: regenerating from unchanged data produces an identical hash; a real data change produces a different one (`tests/core/report-engine.test.ts`).
- **Report validation** (`report-validation.ts`): mechanically enforces "every claim must cite" — any section without `allowEmptyCitations` and zero citations is a critical finding, and `generateReport()` refuses to write a report with critical findings. This caught a real bug pattern during development, repeated across 6 of the 9 report generator files (see below), not just a theoretical one.
- **4 renderers**, pure `ReportData -> string` functions, zero DB access: Markdown, screen HTML, print-ready HTML (page-break-aware, serif, `@page`/`@media print` CSS — the deliberate "PDF-ready HTML" interpretation, since no headless-browser/PDF-binary dependency exists in this environment), and JSON (the `ReportData` itself, so a JSON consumer gets exactly what generated the other three formats).
- **The validation engine's citation check caught a real, repeated bug while building the 9 reports**: six report generators (`workspace_health`, `missing_evidence`, `trademark_readiness`, `priority_of_use_dossier`, `evidence_binder`, `case_summary`) initially rendered gap/finding/risk bullet lists as prose text but passed an empty `citations` array for those sections — the validation engine correctly refused to write every one of them until real `Citation` objects were built from the underlying `EvidenceGap`/`IntegrityIssue`/`DataHealthFinding` rows (new `citeGap()`/`citeIntegrityIssue()` helpers in `report-helpers.ts`). This is exactly what the validation engine is for — it was live-tested against its own real generator bugs, not just a clean-path smoke test.
- **9 real report generators** (`reports/*.ts`), each composing only existing engines: Evidence Engine (`assessWorkspaceEvidence`), evidence-quality-metrics, integrity/data-health/knowledge-validation engines, QueryEngine, Asset Intelligence, the Provenance Engine's `traceResolvedDateProvenance()` (Evidence Binder's chain-of-custody section), and — for Case Summary and case-scoped Evidence Binder — the newly-extracted `composeCaseDetail()`. Priority of Use Dossier's "earliest known evidence per category" deliberately does NOT use `QueryEngine.firstAssetByCategory()` (which orders by raw filesystem `created_at`) — it composes category filtering with real resolved dates instead, correctly applying ADR-010's lesson rather than reintroducing the exact bug it fixed.
- **Delegate, don't duplicate, applied to the API layer itself**: extracted `findConflictingAssets()` (evidence-provenance-engine.ts) and `composeCaseDetail()` (case-builder/case-detail-composer.ts) out of the Phase 4.5 Case Detail API route — the route and Case Summary Report/Evidence Binder now call the same functions instead of three copies of the same joins existing across the codebase. Verified: full test suite and frontend build unchanged after the extraction.
- **A real pre-existing bug found and fixed**: `recordEvidenceGap()` (Phase 3) was a pure append-only INSERT with no equivalent of `resolved_dates`' delete-then-insert discipline (ADR-010) — repeated `assessWorkspaceEvidence()` calls (which report generation is the first caller to make repeatedly in one session) silently accumulated duplicate gap rows forever. Found by inspecting real generated report output (duplicate bullet points in a real Trademark Readiness Report against Fatletic), fixed at the source with a new `clearEvidenceGaps()` called before re-detection, verified against real data and the full test suite.
- **`CaseBuilderService.linkReport()`**: a third public wrapper (alongside `linkAsset`/`linkTimelineEvent`) around the existing generic `link()` method. Verified live: a real Case Summary Report generated for Fatletic's case #1 was linked via `case_links` with `linked_type = 'report'` — the first report row this schema field has ever actually pointed to.
- **CLI**: `generate-report` (`npm run generate-report -- <workspaceId> <reportType> [caseId]`).
- **Real, live verification against both workspaces**: all 9 report types generated successfully against real Fatletic data (up to 633 citations in a single Evidence Binder); all 8 workspace-scoped report types generated successfully against PrecisionWorkz's completely empty evidence tree with zero crashes — real proof of the "missing evidence handling" requirement, not just a fixture test. Cross-workspace leakage checked directly: zero Fatletic asset IDs appear anywhere in PrecisionWorkz's generated exports.
- **Tests**: 14 new (`tests/core/report-engine.test.ts`) covering all 8 required categories from the spec — generation (all 9 types), determinism (hash-stable/hash-changes-on-real-change), citation integrity (structural + asset-existence checks), missing evidence handling (empty workspace, all types), confidence explanations, multi-workspace reports, no cross-workspace leakage, and export validation (all 4 files written, non-empty, JSON round-trips). 156 tests total, all passing.
- **Not built, documented rather than hidden**: no PDF binary generation (no headless-browser dependency added — "PDF-ready HTML" is the literal deliverable); no report caching/incremental-regeneration-skip (unlike Phase 6's vault notes, every report generation call always re-renders and re-writes — reports are explicitly user/CLI-triggered documents, not a continuously-synced view, so this wasn't judged to need Phase 6's skip-if-unchanged mechanism; worth reconsidering if reports end up generated on a schedule); AI, cloud sync, collaboration, graph visualization, attorney/legal-advice logic, and speculative report claims were all correctly out of scope per the phase's own "do not build" list.

## Phase 7 — Import Framework & Plugin Runtime (2026-07-08)

Focused scope, per explicit user direction after the full spec was trimmed: build the production plugin runtime and shared import pipeline in full, but only build extraction logic where it can be validated against real data — never fabricate a parser for an undocumented format. "Build what can be validated. Document what cannot." was the phase's governing rule.

- **Schema:** `migrations/0005_plugin_runtime.sql` — `plugin_registrations` (per-workspace state: discovered/active/disabled/error), `plugin_health` + `plugin_health_events` (a live snapshot plus an append-only log, so "why is this plugin unhealthy" has real history), `import_runs` (the full audit trail: plugin, version, source, counts, validation result, per run — including every folder scan, since scan now runs through this same table).
- **Plugin runtime** (`core/plugin-runtime/`): manifest schema (`plugin-manifest.schema.json`, ajv-validated, same fail-loud pattern as `workspace.schema.json`), a loader (`discoverPluginManifests`, `loadPluginsForWorkspace`) that checks engine/dependency version compatibility and per-workspace activation off `workspace.json` `modules` flags, and one error-isolation choke point (`runPluginCall`) every plugin lifecycle/discover call goes through — success and failure both recorded to `plugin_health`, a thrown plugin exception always arrives at the caller as a `PluginExecutionError`, never a raw crash. Verified live: one plugin's manifest error or a `discover()` throw never stops another plugin from loading or another workspace from scanning (`tests/core/plugin-runtime.test.ts`).
- **Real engine-versus-runtime discovery: `require()` cannot load an arbitrary `.ts` plugin module under vitest.** Confirmed by direct reproduction — `require()` of a real, existing file throws `MODULE_NOT_FOUND` inside Vite's module graph, though it works fine under `tsx`. Switched the loader to dynamic `import()`, which works identically under `tsx`, the `tsc`-compiled build, and `vitest`. Same category of environment-driven substitution as ADR-009 (sql.js) and ADR-011 (local web app) — documented as ADR-012, not silently patched around.
- **Shared import pipeline** (`services/import-pipeline/import-pipeline.ts`): the per-file stage sequence (validate → hash → extract metadata → generate asset ID → timeline candidate generation) and per-run stages (relationship discovery, knowledge update, validation) extracted from the old `import-engine.ts` as a mechanical reorganization, not a rewrite — preserves every byte of previously-tested behavior. "Evidence Extraction" and "Case Update," named in the spec's stage list, are honestly documented as no-ops today (no importer has evidence or case data distinct from metadata/timeline yet) rather than faked.
- **`runScan()` now IS an import**, not a parallel code path: `import-orchestrator.ts`'s `runImport(workspace, pluginId, source, trigger)` is the one general orchestrator; `runScan()` is a two-line wrapper calling it with `importer-generic-folder` and `{kind:"workspace_root"}`. Real dogfooding, not just a claim — every one of this project's tests and every real Fatletic/PrecisionWorkz scan is now live proof the plugin runtime and shared pipeline work correctly, verified by rerunning the full suite (121/121 passed unchanged) and a real Fatletic scan (199 files, 0 created/0 updated — correct, since Fatletic was already fully scanned; the real proof is in the unchanged counts) after the refactor.
- **Added a real Validation pipeline stage**: every import run now ends with a `validateKnowledge()` pass (Phase 3.5's existing engine, delegated to, not duplicated), and its pass/fail result is stored on `import_runs.validation_passed` — a genuinely new capability, not present before this phase.
- **Generic Folder Importer** (`plugins/importer-generic-folder/`): the reference Importer implementation. Its entire `discover()` wraps the existing `scanWorkspaceFiles()` — everything after discovery is the shared pipeline, by construction (the `ImporterPlugin` contract exposes no way to bypass it).
- **ZIP Archive Importer** (`plugins/importer-zip-archive/`): added `adm-zip` (pure-JS, no native compilation — checked against the same constraint that ruled out `better-sqlite3` and Tauri before adding it). Extracts read-only to a new guarded `WorkspaceFs` scratch surface (`.brandos/.import-staging/<runKey>/`, cleaned up via a `cleanup()` hook the orchestrator calls after the pipeline finishes with the files). Every asset's `original_path` is `"<zipRelPath>::<entryName>"` — real citation traceability to exactly which archive and which member, without ever touching the real ZIP. Verified: extraction, provenance-path idempotency (re-import reuses the same Asset IDs), and scratch-directory cleanup, all against a real ZIP built with `adm-zip` in `tests/core/importer-zip-archive.test.ts`.
- **Instagram + Printful plugins, honestly incomplete**: real manifests, real registration, real per-workspace activation (verified: both `active` in Fatletic, both `disabled` with a specific reason in PrecisionWorkz), real compatibility checking, real error-isolated `discover()` calls — that all throw a clear, attributed "blocked on real export data" error rather than fabricating extraction logic or silently returning zero files. Each plugin ships a `BLOCKED.md` stating exactly what's done, what's missing, and what unblocks it. No Instagram export and no structured Printful export (only PDF invoices, already imported as ordinary evidence) exist anywhere in this repository — checked before writing a single line of parsing code.
- **CLI**: `import-zip`, `list-plugins`, `list-imports`, `validate-golden-dataset` — all wired into `package.json`.
- **Golden Dataset** (`tests/golden-dataset/build.ts`): a deterministic builder (not a binary fixture committed to git) producing a fixed, known-shape evidence set — a source→export pair, a duplicate pair, a real minimal PDF. `GOLDEN_EXPECTATIONS` is the permanent regression baseline: asset counts, relationships, duplicates, timeline resolution, knowledge validation, vault generation, case integrity, and import idempotency are all asserted against it, both as `tests/core/golden-dataset.test.ts` (8 tests) and as a standalone CLI gate (`npm run validate-golden-dataset`) that doesn't require the test runner.
- **Plugin SDK** (`docs/PLUGIN_SDK.md`): interfaces, manifest schema, lifecycle, the dynamic-`import()` decision explained, capabilities, events, configuration (honestly stated as minimal — no plugin has needed more than `modules` flags yet), error handling, logging, testing checklist, and the Generic Folder Importer as the fully worked example.
- **Housekeeping**: removed `import-engine.ts`'s dead `watchWorkspace()` function (Phase 5 audit finding #2, zero call sites confirmed by grep before deletion) while already rewriting that file — the natural point Phase 5's own review recommended resolving it.
- **Tests**: 20 new (8 golden dataset, 8 plugin runtime, 4 ZIP importer) — 141 total, all passing. Backend `tsc --noEmit` clean, frontend `tsc --noEmit` + `vite build` clean (unchanged — Phase 7 touched no frontend code).
- **Not built, documented rather than hidden**: import cancellation/resume/rollback (the two real importers complete synchronously in seconds against real data; building a resumable-job state machine for that would be exactly the speculative infrastructure this project's principles argue against — see `ARCHITECTURE_DECISIONS.md` ADR-012), `deactivate()` has no caller yet (no workspace-closing lifecycle event exists), and no runtime support for `Classifier`/`ReportTemplate`/`VaultTemplate` plugin types (no second real implementation of any of them exists to validate a loader against, same reasoning that deferred the Importer loader itself across 5 prior phases).

## Phase 6 — Knowledge Layer & Obsidian Integration (2026-07-08)

BrandOS's first generated-documentation layer. Architecture principle enforced throughout: BrandOS owns the data, Obsidian is a generated view — nothing here makes BrandOS depend on Obsidian, and nothing invents a fact the engine hasn't already validated.

- **Schema:** `migrations/0004_obsidian_vault.sql` — `obsidian_notes` (content-hash + manual-edit tracking per generated file, the mechanism behind incremental regeneration and edit preservation).
- **`WorkspaceFs` extended** with a second guarded write surface (`writeVaultFile`/`readVaultFile`), scoped to the workspace's Obsidian vault directory exactly like `.brandos/` — structurally impossible to write anywhere else, same pattern as the evidence-tree guard since Phase 2.
- **Vault generator** (`services/vault-generator/`): note templates for Assets, Cases, and the Workspace, built entirely from existing engine aggregators (`AssetIntelligenceView`, `CaseBuilderService`, `computeEvidenceQualityMetrics`) — no new data logic, only Markdown formatting. "Summary" fields are templated sentences assembled from real fields, never generated prose.
- **Edit preservation, implemented per ADR-008's open question — resolved this phase in favor of delimited generated-block markers** (`<!-- brandos:generated:start/end -->`) over sibling files: keeps one file per entity, the better Obsidian UX. Content below the closing marker is the user's own and is never touched; content inside the markers is regenerated only if it actually changed, and never overwritten if a hand-edit to the generated block itself is detected (content-hash mismatch) — verified on real Fatletic data, not just fixtures (see below).
- **Incremental regeneration** (Section 6): every note's generated-block hash is compared before any write; unchanged notes are skipped entirely. Proven at real scale: after a small engine-layer fix (see below), regenerating Fatletic's 213-note vault correctly updated only the 199 asset notes whose content actually changed and skipped the 14 case/workspace/index notes that didn't — zero false positives.
- **Index pages, not duplicate entities:** "Products," "Designs," "Logos," "Campaigns," etc. aren't real database entities — they're real classification/tag queries rendered as link lists to the *one* canonical note each asset already has under `Assets/`. No asset is ever duplicated into two files (a documented architecture decision, not an oversight).
- **Living Knowledge Review** (`knowledge-review.ts`, Section 9): real checks against the generated vault + database — missing notes, orphaned notes, broken links, stale content (an asset rescanned after its note was last generated), and unresolved DB references (delegates to Phase 3.5's `validateKnowledge()` rather than re-implementing reference-integrity checks).
- **A real bug found and fixed while generating real content, not shipped silently:** `AssetIntelligenceView` (built in Phase 3, consumed by Mission Control's Asset Detail page since Phase 4.5) never included the Phase 3.5 resolved-date — meaning both the vault's first-draft asset notes *and* the live dashboard were displaying the raw `file_created` timeline event (in Fatletic's case, literally `1970-01-01`, labeled "confidence 100") as if it were a trustworthy fact, silently bypassing the entire multi-candidate date-resolution system Phase 3.5 built. Fixed at the source (`asset-intelligence.ts` now includes `resolvedDate`) so the fix applies to both the vault and Mission Control simultaneously — the Asset Detail page now shows "Resolved Date" (with real reasoning) as the primary answer and raw candidate events as a clearly-labeled, subordinate list. Regenerating the real Fatletic vault after the fix correctly re-wrote exactly the 199 asset notes affected.
- **CLI:** `npm run generate-vault -- <workspaceId>`.
- **Tests:** 10 new (121 total, up from 111) covering real note generation, incremental skip/update behavior, both edit-preservation scenarios (content below the marker; a hand-edit to the generated block itself), and the Living Knowledge Review's real checks.
- **Real generation, both workspaces:** Fatletic — 213 notes (199 assets + 5 cases + 1 workspace + 8 indexes), 0 critical/warning Living Knowledge Review findings, 11 info-level. PrecisionWorkz — 9 notes (1 workspace + 8 empty indexes), correctly isolated from Fatletic's vault (Section 11), zero cross-contamination. A real hand-edit to a real Fatletic note (content below the marker) was written, confirmed preserved through a real regeneration, then cleaned up afterward so it doesn't sit in real data as if it were genuine curation.
- **Section 10 (performance target: 100,000+ notes) — addressed honestly, not benchmarked with synthetic data:** the incremental-regeneration mechanism (hash-skip unchanged notes) is architecturally the right shape for that scale, but no vault of that size was generated to prove it empirically, since doing so would mean fabricating 100,000 fake notes purely to produce an impressive-looking number — exactly what this phase's own "never invent facts" principle argues against. Stated plainly in `PHASE_6_KNOWLEDGE_LAYER_VALIDATION_REPORT.md` rather than implied to be tested.

## Phase 5 — Platform Consolidation (2026-07-08)

Engineering-quality phase, not a feature phase — deliberately trimmed by explicit user direction to four objectives: shared type system, multi-workspace verification, an honest architecture/code audit, and a platform validation report. Explicitly out of scope this phase: API client generation, developer-doc expansion, lint/format tooling, cosmetic refactors, speculative optimization.

- **Shared type system** (`app/shared-types/index.ts`, new): the single source of truth for every API request/response shape (`WorkspaceSummary`, `OverviewData`, `CaseSummary`/`CaseDetail`/`CaseTemplate`/`CaseEvidenceSuggestion`, `AssetSummary`/`AssetIntelligence`/`AssetFacets`, `PriorityOfUseData`, `SearchResult`, `ActivityEvent`, `ActionItem`, `ReviewQueueEntry`, `DuplicateGroup`). Both `app/src/api/routes.ts` (every handler now constructs an explicitly-typed response object) and `app/web/src/api.ts` (re-exports from shared-types instead of declaring its own copies) import from here — the exact fix for the type-duplication risk documented in Phase 4 and confirmed as a real incident in Phase 4.5. Wired via a plain relative-import folder (no build step, no new package.json, no npm workspaces) — `app/tsconfig.json`'s `include` and `app/web/vite.config.ts`'s `server.fs.allow` both updated to reach the shared location. Verified end-to-end: both packages typecheck clean, frontend production build unchanged in size (confirms type-only imports are fully erased, zero runtime cost), and the Vite dev server correctly serves the shared file (`fs.allow` fix confirmed via direct HTTP request, not assumed).
- **Multi-workspace isolation, finally verified for real** (deferred across Phases 3.5, 4, and 4.5): a new automated test suite (`tests/core/multi-workspace-isolation.test.ts`, 5 tests) proves two independently-scanned fixture workspaces share nothing — separate database files, zero asset leakage, zero case leakage, zero search-result leakage, and correctly independent duplicate-group/review-queue detection even when both workspaces contain byte-identical content (the scenario most likely to falsely merge them if isolation were broken). Beyond fixtures: PrecisionWorkz was scanned and exercised through the **actual API layer for the first time** — real HTTP requests against a live server confirmed its overview, cases, and search all correctly return empty/isolated results while Fatletic's real 5 cases and 198 real search results remained simultaneously unaffected in the same running process (the real test of `db-cache`'s per-workspace connection isolation, not just a theoretical one).
- **Architecture & code audit** — real findings, documented not auto-fixed (per explicit instruction): duplicated row-mapping logic (`mapAssetRow` in `repositories.ts` and `mapAsset` in `query-engine.ts` independently implement the same snake_case→camelCase conversion), two genuinely dead functions (`CaseBuilderService.createCustom()` and `import-engine.ts`'s `watchWorkspace()`, both zero-usages-found outside their own definitions), and four scaffolding directories empty since Phase 0/1b with zero content across 5+ phases (`app/src/plugins/`, `app/shared/`, `app/scripts/`, `app/obsidian-temple/`). Also confirmed clean: no unused npm dependencies, no circular dependencies between `core/` and `api/`, no workspace/brand literal leakage in any TypeScript source (only a factual code-comment reference), all 5 CLI scripts correctly wired to `package.json`. Full prioritized findings in `PHASE_5_PLATFORM_VALIDATION_REPORT.md`.
- **Tests:** 111 total (up from 106), all passing. Both packages typecheck and build clean throughout.
- **Docs:** this entry, `ARCHITECTURE_REVIEW.md` Phase 5 section, `ARCHITECTURE_DECISIONS.md` Phase 5 consistency note (no new ADR — Section 1 implements existing principles, doesn't change them), `IMPLEMENTATION_PLAN.md`, and `PHASE_5_PLATFORM_VALIDATION_REPORT.md` (new).

## Phase 4.5 — Mission Control Evolution (2026-07-08)

Explicit rule carried forward and re-verified: the dashboard still contains no business logic. Every new value traces to an existing or newly-added *engine-layer* function; nothing is computed in a route handler or a React component.

- **New engine capabilities (small, defensible, core layer):**
  - `activity-feed-engine.ts` — a read-time UNION over `scan_runs`, `knowledge_validation_runs`, `cases`, `case_links`, `relationships`, and `data_health_findings`, producing one chronological feed. No new storage.
  - `QueryEngine.listAssetsFiltered()` — general-purpose classification/tag/extension/needs-review filtering + sort, the Evidence Explorer's data source, written to be reusable by any future consumer.
  - `CaseBuilderService.suggestUnlinkedEvidence()` — groups unlinked assets by tag for empty-state guidance, explicitly framed as browse candidates, never a claim that they support the case (ARCHITECTURE_PRINCIPLES.md #4).
- **New API routes:** `/activity`, `/action-center` (pure composition of existing counts/gaps into a clickable list), `/review-queue` (enriched drill-down), `/duplicates` (enriched drill-down), `/cases/:id/suggestions`, and `/assets` upgraded to support real filtering + facet counts.
- **Information hierarchy (Overview redesign):** hero tier (Trademark Readiness, Priority of Use, Needs Review, Workspace Status, Evidence Quality) rendered large; mid tier (Timeline, Relationships, Cases, Health) at normal weight; low tier (Architecture Health, Duplicate Groups) small and quiet at the page bottom — via typography/sizing/layout only, no new data.
- **New pages:** Asset Detail (closes the gap flagged in Phase 4's own review — fills in classification, tags, a relationship "knowledge navigation" panel, timeline, metadata, linked cases, hash), Needs Review drill-down, Duplicate Groups drill-down, Evidence Explorer (filter by classification/tag, sort, with real facet counts).
- **Case Workspace enrichment:** Readiness/Confidence hero row, empty-state evidence suggestions (tag-grouped, neutral framing), honest Notes placeholder ("not yet supported" — no fake feature), honest Exports placeholder ("planned for a future phase").
- **Search grouping (Section 7):** Cmd+K results now grouped by entity type (Cases, Assets, Timeline Events, Tags) in priority order; selecting an asset result now correctly opens its real Asset Detail page.
- **Two real bugs found and fixed during this phase, not shipped silently:**

## Phase 3.5 — Evidence Reliability & Knowledge Validation (2026-07-07)

No UI, dashboard, Obsidian, reports, AI, exports, or reorganization — purely correctness/trustworthiness work on what already existed. Fixes the Phase 3 epoch-timestamp finding architecturally (ADR-010), not with a patch.

- **Schema:** `migrations/0003_evidence_reliability.sql` — 6 new tables (`candidate_dates`, `date_source_priorities` seeded with the specified default order, `resolved_dates`, `data_health_findings`, `knowledge_validation_runs`) plus explainability columns added to `classifications` and `review_queue`.
- **Timeline Intelligence** (`services/timeline-intelligence/`): collects candidates from filesystem, EXIF/PDF/PSD/XCF metadata, filename patterns, folder patterns, and relationship inference; plausibility-checks every one (epoch/pre-1990/future rejected, never deleted, always explained why); the Resolution Engine picks a winner by configurable priority among plausible candidates only, with full reasoning and every rejected alternative recorded.
- **Evidence Provenance Engine**: traces a resolved date through its winning candidate to a hash-verified original asset and the workspace — the concrete, currently-buildable slice of the full Trademark-Readiness-to-hash chain described in the spec.
- **Classification improvements**: new known-product-folder-structure rule (real signal from Fatletic's actual "Clothing Promo" folder tree), sibling-majority context boost for the generic low-confidence "Image" fallback — both deterministic, no AI.
- **Needs Review Intelligence**: every review-queue entry now carries a concrete suggested action, estimated effort, and potential impact, computed from real duplicate/relationship/case-link counts, not invented.
- **Data Health Engine**: new epoch/implausible-date, broken-candidate-provenance, and duplicate-metadata checks — delegates to Phase 3's Integrity Engine for the checks that already existed rather than reimplementing them.
- **Knowledge Validation Engine**: 9-point pass/fail checklist (every asset has an ID, every relationship/timeline/case-link/report references a real asset, every active asset is hashed, every resolved date references a real candidate, no duplicated metadata, provenance chains complete).
- **Evidence Quality Metrics**: 11 named, formula-documented metrics (timeline/metadata/relationship/classification completeness, confidence distribution, needs-review %, duplicate/evidence/provenance coverage, health score, missing/conflicting evidence scores) — one reusable function, not one-off queries per future consumer.
- **Wired into the scan pipeline:** candidate collection + date resolution run on every new/changed asset, plus a relationship-derived second pass (an export can inherit a candidate from its resolved source file). Classification/tagging upgraded to call the new explainability path.
- **CLI:** `analyze` extended with Data Health, Knowledge Validation, and Evidence Quality Metrics output (replacing the plain Phase 3 integrity-check section, which is now folded into Data Health rather than run twice).
- **Tests:** 30 new tests (105 total, up from 75) across 5 new files. Two real bugs found and fixed via testing: an ambiguous `asset_id` column reference in two SQL joins (the `assets` table has both an integer `id` and a string `asset_id` column — a genuine SQL bug, not a test artifact), and a resolution-engine gap where re-resolving to "no plausible candidates" left a stale prior resolution in place instead of clearing it.
- **Real run against Fatletic** (199 real assets, fresh scan): confirmed all 199 `filesystem_created` candidates correctly excluded as epoch artifacts; system fell through to `filesystem_modified` (185), real PDF metadata (11), and EXIF (3); average resolved confidence 52.1/100 (honest, not the prior false 100). Classification completeness rose from 36% (Phase 3) to 73%, with the generic "Image" bucket dropping from 128 to 53 assets as 50 correctly reclassified to a new "Product Photo" category and Export detection improved from 3 to 26. Full numbers in `app/docs/PHASE_3_5_VALIDATION_REPORT.md`.
- **New finding, disclosed not hidden:** continuous-use score remains weak (1/100) even at 100% timeline completeness, because the `filesystem_modified` fallback — while plausible — isn't independently verified as accurate for continuous-use purposes; and the health-score formula proved miscalibrated on real data (hit 0/100 from finding *volume*, not severity). Both documented as open items, not smoothed over.

## Phase 3 — Knowledge Layer (2026-07-07)

No UI, no dashboard, no Obsidian generation, no reports, no AI classification — all explicitly out of scope and not built. Added `app/docs/ARCHITECTURE_PRINCIPLES.md` (the project's constitution) and `app/docs/ARCHITECTURE_REVIEW.md` (new, cumulative, updated after every phase from here on).

- **Schema:** `migrations/0002_knowledge_layer.sql` — 9 new tables (`tags`, `asset_tags`, `classifications`, `review_queue`, `evidence_assessments`, `evidence_gaps`, `case_templates` seeded with the 8 named templates, `asset_notes`, `integrity_checks`). Every new table references `assets.id` or another table's id — nothing duplicates a fact that lives elsewhere.
- **Rule-based classification** (`services/classification-engine/`): deterministic rules, no AI. Path-context rules (Instagram/Printful/proof/invoice/customer/legal keywords) run before extension rules, since folder context is a stronger signal than file type alone. Confidence reflects real signal strength — a PSD's own binary header scores 95, a bare image with no relationship or path context scores 65 and lands in Needs Review.
- **Tag engine** (`services/tag-engine/`): deterministic tag suggestions from classification category + path keywords, every suggestion carries a stated reason.
- **Evidence engine** (`services/evidence-engine/`): completeness, continuous-use, and priority-of-use scores computed by plainly-documented formulas over real classification/timeline rows — every assessment's `notes` field states exactly which counts produced the score. Detects and records evidence gaps (e.g. "no dated evidence for Marketing Evidence category").
- **Case Builder** (`services/case-builder/`): full service — create from template or custom, link/unlink assets and timeline events by reference only, case-scoped evidence-strength assessment (delegates to the evidence engine, no separate scoring logic).
- **Query engine** (`services/query-engine/`): discrete typed methods (first-by-category, every-by-extension, before-date, connected-to, missing-metadata, duplicate-groups, orphaned, by-tag, needing-review) — a reusable foundation, not a NL query parser (explicitly future scope).
- **Graph engine** (`services/graph-engine/`): a read-time view over relationships/case_links/asset_tags/timeline_events — no separate graph storage, per Principle #8.
- **Provenance engine** (`services/provenance-engine/`): cycle-safe BFS both directions over the relationships graph, builds an upstream→self→downstream chain for any asset.
- **Integrity engine** (`services/integrity-engine/`): detects broken references, hash mismatches, missing/duplicate/orphaned assets, circular relationships, all persisted to `integrity_checks`.
- **Search foundation** (`services/search-engine/`): unified LIKE-based search across filenames, paths, asset IDs, hashes, tags, metadata values, case titles, timeline titles — explicitly not FTS5/semantic yet (future scope), but swappable behind one function without touching callers.
- **Asset Intelligence** (`services/asset-intelligence/`): the aggregator — composes classification, tags, relationships, timeline, provenance, linked cases, and notes for one asset by Asset ID, entirely from existing normalized tables (no new asset columns, no duplicated storage).
- **Wired into the scan pipeline:** `import-engine.ts` now classifies and tags every new/changed asset after relationship detection (so "is this image an Export?" can see real relationship data), plus catches up any asset from a prior scan that has no classification yet. Four new events (`classification.assigned`, `tags.assigned`, `evidence.assessed`, `integrity.issue_found`).
- **CLI:** `npm run analyze -- <workspaceId>` (runs evidence assessment + integrity check, prints a summary).
- **Tests:** 42 new tests (75 total, up from 33) across 6 new files covering every engine, all against real scan output (not mocked data).
- **Real run against Fatletic** (199 real assets): 128 Image, 25 Design Source, 18 Documentation, 16 Commerce Evidence, 9 Historical Evidence, 3 Export. Completeness 36/100 (71 of 199 confidently classified — the rest correctly land in Needs Review rather than being guessed). Tag counts: Brand/Logo 109 each, Final 80, Product 61, Evidence 25, Prototype 25, Historical 9, Invoice 7, Unused 5. Integrity check: 58 issues, all info-level (11 duplicate groups, 47 orphaned assets) — zero critical/warning issues (no broken references, no hash mismatches, no circular relationships).
- **Significant finding, not a code bug:** every one of the 199 timeline events (100%) came back with a `file_created` date of exactly `1970-01-01` (Unix epoch). This tanked the continuous-use score to 1/100. Root cause: WSL's `drvfs` mount (mounting the Windows C: drive into Linux) does not reliably populate `birthtime` — Node's `fs.Stats.birthtime` falls back to epoch zero. This is an environment/filesystem limitation, not a BrandOS bug, but it means **`created_at` is currently not a trustworthy evidence signal on this setup** — a real risk for a system whose whole purpose includes Priority of Use dating. Documented in full in `ARCHITECTURE_REVIEW.md` with a recommended fix (detect epoch-or-near-epoch birthtime and fall back to `modified_at`, flagged as inferred not verified) for a future phase — not fixed in this one, since it's a Phase 2 ingestion concern outside Phase 3's declared knowledge-layer scope, and is disclosed here rather than silently patched.
